# Jumerra

## Overview

Jumerra is an AI-powered talent ecosystem designed to intelligently connect early-career candidates (interns, graduates) with employers. It also provides educational institutions with real-time student placement tracking. The project aims to streamline entry-level recruitment, offer valuable insights into talent pipelines, and analyze hiring trends.

## System Architecture

Jumerra is structured as a monorepo using pnpm workspaces. The API specification, defined in `lib/api-spec/openapi.yaml`, acts as the single source of truth for API contracts and Zod validator generation.

**Core Components:**
- **API Server**: An Express 5 application leveraging Drizzle ORM, pino for structured logging, and domain-split routes. It encapsulates business logic, including the matching algorithm.
- **Database**: PostgreSQL, managed by Drizzle ORM with a domain-split schema.
- **API Client**: React Query hooks generated by Orval from the OpenAPI spec.
- **Web App (`artifacts/talent-platform`)**: A React application built with Vite, using wouter for routing, shadcn/ui for components, Recharts for data visualization, framer-motion for animations, and sonner for toasts. Features a real cookie-session authentication system with a "View as" demo fallback.
- **Mobile App (`artifacts/talent-mobile`)**: An Expo and React Native application for candidates, reusing the web app's API client and backend. Includes its own candidate-only sign-in and sign-up screens (route group `(auth)`) gated by an `AuthGate` provider that redirects unauthenticated users to sign-in and authenticated users into the tabs. Uses Feather icons (not lucide). Shared `customFetch` sends `credentials: "include"` so cookie sessions work for the cross-origin Expo web preview during development. On native (iOS/Android Expo Go), React Native's `fetch` does not maintain a cookie store between calls — especially on Android — so `app/_layout.tsx` installs an AsyncStorage-backed `CookieJar` (registered via `setCookieJar` from `@workspace/api-client-react`) that captures `Set-Cookie` from responses and re-attaches the `Cookie` header to subsequent requests. Without this, login appears to "succeed" (200) but the session cookie is dropped, so `/auth/me` returns null and `AuthGate` silently bounces the user back to sign-in.
- **Duplicate-email signup UX**: For the candidate role only, `POST /auth/register` returns `409 { error: "An account with this email already exists. Please sign in instead." }` for both pre-checked duplicates and the racy unique-constraint violation (`23505`) caught in the catch block. The mobile sign-up screen detects status `409`, shows the message inline, and renders a "Sign in instead" link that hands off to `(auth)/sign-in` with `?email=` prefilled (read via `useLocalSearchParams`). Employer/institution signups still use the silenced 201 response to prevent email enumeration on org accounts (they don't auto-login, so the user-stuck UX problem doesn't apply there).
- **Auth endpoints disable HTTP caching and strip conditional headers**: The `/auth/*` Express router (i) deletes `If-None-Match`/`If-Modified-Since` from every incoming request so Express's freshness check can never short-circuit to a body-less `304`, and (ii) sets `Cache-Control: no-store, no-cache, must-revalidate, private` (+ legacy `Pragma`/`Expires`) on every outgoing response so well-behaved clients never cache auth state in the first place. This was needed because iOS NSURLSession and browser HTTP caches both store GET responses with their ETag, and the first `/auth/me` after app launch returns `{user: null}`. After sign-in the next `/auth/me` would re-use the same `If-None-Match`, and if the server still computed `{user: null}` (cookie not yet attached, cross-site SameSite block, etc.) Express returned `304` with no body. Our shared `customFetch` returns `null` for any 304 → `useAuth` saw no user → `AuthGate` bounced the candidate back to sign-in (perceived as "wrong credentials"). Stripping the validators on the request side makes the fix work even for clients with stale caches from before the deploy.
- **CORS is locked to an explicit allowlist**: `artifacts/api-server/src/app.ts` no longer reflects arbitrary origins. The allowlist is built from `https://$REPLIT_DEV_DOMAIN`, `https://$REPLIT_EXPO_DEV_DOMAIN`, every entry in `REPLIT_DOMAINS` (production deploys), and `localhost`/`127.0.0.1` (any port). This is required because the session cookie is now `SameSite=None; Secure` (see below); a wildcard `Access-Control-Allow-Origin` combined with `Access-Control-Allow-Credentials: true` would let any third-party site call our authenticated API and read the response. Requests with no `Origin` header (curl, native iOS/Android Expo Go) pass through untouched — there's no CSRF surface there because they aren't browser contexts. Disallowed browser origins still get a normal `200`/whatever status, but the response carries no `Access-Control-Allow-*` headers, so JS in the attacker's page cannot read it.
- **Session cookie is also `Partitioned` (CHIPS)**: The Replit workspace preview pane shows the mobile app inside a cross-origin iframe (parent = workspace, iframe src = Expo dev domain, API on yet another subdomain). Modern Chrome blocks "third-party cookies" in iframe contexts even when they have `SameSite=None; Secure` — so the post-login `GET /auth/me` would arrive without a cookie, AuthGate would see `{user: null}`, and the user would see a brief flash and bounce back to the sign-in screen. The fix is the [`Partitioned` cookie attribute (CHIPS)](https://developer.mozilla.org/en-US/docs/Web/Privacy/Privacy_sandbox/Partitioned_cookies). `express-session@1.19` doesn't surface the option, so `artifacts/api-server/src/lib/session.ts` exports `sessionPartitionedCookiePatch()`, a tiny middleware (mounted in `app.ts` immediately before the session middleware) that wraps `res.setHeader` and appends `; Partitioned` to any `Set-Cookie` line whose cookie name is `talentlink.sid`. The cookie now reads: `talentlink.sid=...; Path=/; HttpOnly; Secure; SameSite=None; Partitioned`. Cookies are partitioned by the **top-level site**, so each top-level context (workspace iframe, direct page load, etc.) has its own partition — direct first-party logins to the talent-platform web app are unaffected.
- **Session cookie is `SameSite=None; Secure`**: The mobile app's Expo web preview is served from `$REPLIT_EXPO_DEV_DOMAIN` while the API is on `$REPLIT_DEV_DOMAIN` — different hosts, so requests from the preview to `/api/*` are cross-site. Modern browsers refuse to send `SameSite=Lax` cookies on cross-site sub-requests, so a successful `POST /auth/login` would set the session cookie but the immediate cross-site `GET /auth/me` would not include it, leaving the AuthGate stuck. The cookie is now `SameSite=None; Secure` (both required by spec). The Replit proxy already serves everything over HTTPS and sets `X-Forwarded-Proto`, so `Secure=true` is honored in development too via `app.set("trust proxy", 1)`. The opt-out env var `SESSION_COOKIE_SECURE=false` remains supported for local non-HTTPS work.

**Key Architectural Decisions:**
- **Authentication**: `express-session` with `connect-pg-simple` manages sessions. User roles (`candidate`, `employer`, `institution`, `admin`) are central, augmented by `org_role` for in-organization permissions. bcrypt handles secure passwords.
- **Admin Console**: A dedicated dashboard for user management, application oversight, and analytics. Features include account status toggling, password reset links, and account manager assignments.
- **Customizable Roles & Permissions**: A unified system for admin, employer, and institution roles, with dynamic permission sets managed via database tables and unique indexing for scope+org. Permissions are enforced server-side via middleware and client-side for UI filtering.
- **Candidate Onboarding & Notifications**: Candidates self-register, while employers and institutions follow an approval flow. Institutions can verify candidate affiliations, triggering in-app notifications stored in a `notifications` table and displayed via a bell icon.
- **Institution Management**: Owners can manage their organization's profile, departments, and facilities via the dashboard. Institution types are categorized by an enum. Data is stored in `institution_departments` and `institution_facilities` tables.
- **Website Builder**: A key-value store (`site_content`) allows administrators to manage public-facing content with an admin-only bulk upsert API.
- **Universal User Profile & Avatar Upload**: All user types share a common profile page with fields for `fullName`, `phone`, `title`, `bio`, and `avatar_url`. Avatar uploads utilize presigned URLs for direct object storage interaction.
- **Matching Algorithm**: Employs a scoring system based on skill coverage (65%), experience (15%), and talent score (20%).
- **Payments (Paystack primary + Stripe fallback)**: Checkout is routed by currency in `artifacts/api-server/src/lib/payment-rail.ts` — `NGN`/`GHS`/`ZAR`/`KES` go to Paystack (when `PAYSTACK_SECRET_KEY` is set), everything else to Stripe. The five payment tables (`boost_payments`, `cv_payments`, `job_tier_payments`, `institution_subscriptions`, `employer_subscriptions`) each carry `provider` (`'stripe' | 'paystack'`) and a nullable `paystack_reference` so a single table can hold rows from both rails; for Paystack rows the existing `stripe_session_id` UNIQUE column re-uses the Paystack reference so we keep one source of truth for "external payment id". Webhook deliveries land at `/api/webhooks/stripe` and `/api/webhooks/paystack`, both mounted in `app.ts` with `express.raw({type:'application/json'})` BEFORE `express.json()` so signature verification gets the exact bytes the provider signed. Every event is deduped via `webhook_events` (UNIQUE on `(provider, event_id)`) by `processWebhookOnce`, which catches the duplicate `23505` and short-circuits — Stripe and Paystack both retry aggressively and this is the only thing standing between us and double-unlocking a candidate. Each successful event is handed to a shared finalizer in `lib/payment-finalizers.ts` that runs the status flip + the unlock work (boost expiry stacking, `aiCvUnlocked` flag, job-tier upgrade, subscription period end) inside a single DB transaction; the CV `/verify` endpoint was previously doing the flip + unlock as two sequential queries and is now also routed through the same finalizer so a crash between the two writes can no longer leave a paid CV payment without its corresponding unlock. The webhook routes are mounted at the very top of `routes/index.ts` (before any auth gate) because they authenticate themselves via signed-payload HMAC, not session cookies. **Deferred as drift (next task):** admin payments console UI, customer-facing rail picker UI, Paystack rail wired into CV/job-tier/subscription checkout endpoints (today only boost has the full Paystack branch; CV/job-tier/subs still create Stripe sessions but their webhooks already accept Paystack `charge.success` events for any row that has a `paystack_reference`).
- **UI/UX**: Features a clean, modern aesthetic with shadcn/ui components, an emerald/teal color palette, dark mode, and lucide icons. No emojis are used.
- **Data Model**: Core entities include `institutions`, `employers`, `candidates`, `jobs`, and `applications`, with detailed sub-models and `candidate_institutions` for managing affiliations.
- **Per-Department Scoping**: Institution staff can be assigned to specific departments, limiting their access to candidates within those departments. This scoping applies to candidate verification, management, and staff invitations.
- **Faculty Layer & Role-Based Institution Access**: Institutions whose `type` supports it (everything except SHS) now have an explicit Faculty layer above departments (`institution_faculties` table; `institution_departments.faculty_id` FK). Five org roles control institution access: **owner** and **registrar** are owner-equivalent (implicit-all, manage everything academic — owner additionally manages subscription/billing); **dean** is faculty-scoped via `users.assigned_faculty_id` and sees students across every department under that faculty; **HoD** is department-scoped via `users.assigned_department_id`; **coordinator/viewer** are unscoped readers. Authorization is enforced in `routes/institutions.ts` (`/institutions/:id/students` requires same-org auth + `students:view` perm; deans/HoDs whose scope row was deleted via FK SET NULL are explicitly denied rather than silently broadened to org-wide) and in `canActOnCandidateAffiliation` (used by verify/unverify), with the same scope-null guard. Candidates pick their faculty + department per affiliation through a cascading picker on both the web profile (`/account/profile`) and the mobile profile-edit screen — the faculty picker is hidden for SHS-type institutions, the department list is filtered by the chosen faculty, and the saved `facultyId` is derived server-side from the chosen `departmentId` (the affiliation API only persists `departmentId`).

- **Email (Resend) + observability**: Transactional email is delivered via Resend through a single dispatcher in `artifacts/api-server/src/lib/email.ts` (auth setup/reset, weekly engagement digest, endorsement, fast-track SLA, hire notifications to institution owners/registrars, registration approval/rejection). When `RESEND_API_KEY` is unset the dispatcher logs a redacted line and returns `{ sent: false, reason: "email-not-configured" }` so callers (admin onboarding, forgot-password) can surface the link as a copy-paste fallback. Every send goes through `lib/email-templates.ts` (`renderEmailHtml`), an inline-CSS wrapper using the teal #0d9488 brand so any visual tweak is a one-file change. Logging policy: reset links are NEVER logged in plaintext (would be a single-step takeover via log access); only kind + a short token fingerprint is logged. Setup links MAY be logged because the admin UI also surfaces them. Optional env vars: `EMAIL_DEFAULT_FROM` (default `Jumerra <onboarding@resend.dev>`), `EMAIL_REPLY_TO`. Server-side error reporting via Sentry (`@sentry/node`) is initialized in `lib/sentry-server.ts` only when `SENTRY_DSN_SERVER` is set — silently no-op in dev so local runs don't ship events. PII is scrubbed in `beforeSend` (authorization/cookie headers, request body, user.email, user.ip_address). Web Sentry (`@sentry/react`) lives in `artifacts/talent-platform/src/lib/sentry.ts` and is gated on `VITE_SENTRY_DSN_WEB`. `@sentry/*` is externalized in `artifacts/api-server/build.mjs` because Sentry's Node SDK dynamically requires several `@opentelemetry/*` instrumentation packages — bundling it emits hard imports that fail at runtime since `@opentelemetry/*` are already external. `@workspace/db` declares `@opentelemetry/api` as a direct dep so its drizzle-orm resolves to the same peer-virtual variant as api-server's (avoids a tsc "two drizzle-orm types" error). Boot-time env validator in `lib/env-validator.ts` fatals (exit 1) on missing required vars in production (`DATABASE_URL`, `SESSION_SECRET`) and warns on missing optional vars (Resend, Stripe, Paystack, Sentry, object storage).
- **Request-id propagation + PII redaction**: `middleware/request-id.ts` generates a ULID per request (honors a sanitised incoming `x-request-id` for end-to-end tracing), attaches it to `req.id`, and echoes it back as the `x-request-id` response header (exposed via CORS so browser JS can read it). pino-http picks it up via `genReqId` so every server log line is correlated. The shared `customFetch` (`lib/api-client-react/src/custom-fetch.ts`) reads it off the response and surfaces it as `ApiError.requestId` so toasts / Sentry breadcrumbs can show a single id that grep'd against the server log. Pino's `redact.paths` list scrubs auth headers (`authorization`, `cookie`, `set-cookie`, `x-api-key`) and PII fields (`email`, `phone`, `password*`, `token`, `bearer`) at the top level and at one/two levels of nesting (`*.email`, `*.*.email`, etc.) — pino does not support deep wildcards so the explicit per-depth list is intentional. The unhandled-error JSON renderer in `app.ts` includes the request id in the 500 body for the user to share with support.
- **Health endpoints**: Split into `/api/healthz` (liveness — always 200 once the process is up; no external deps touched so an unhealthy DB doesn't cause restart loops) and `/api/readyz` (readiness — DB ping via `select 1` + records which payment rails + email are configured; returns 503 with a per-check breakdown when degraded so a load balancer can de-pool the instance without tearing it down).
- **List API hardening (cursor pagination + SQL filters + rate limits)**: The four hot read paths — `GET /candidates`, `GET /jobs`, `GET /applications`, `GET /institutions/:id/students` — were rewritten to (i) accept `?limit=` (default 20, **server-capped at 50**) and `?cursor=` opaque base64-url cursors, (ii) push **every** filter into the SQL `WHERE` instead of `Array.prototype.filter()` over the entire table, and (iii) return the next-page handle in the **`X-Next-Cursor` response header** (CORS-exposed). The header surface — not an envelope `{items,nextCursor}` — was a deliberate drift call so we didn't have to migrate 30+ web/mobile call-sites; the response body stays a flat array. Helpers live in `artifacts/api-server/src/lib/pagination.ts` (`parseLimit`, `encodeCursor`, `decodeCursor`, `setNextCursor`). Cursor shapes per endpoint match the ORDER BY tuple lexicographically: candidates `(isBoosted, talentScore, id)`, jobs `(tierRank, featured, postedAt, id)`, applications `(appliedAt, id)`, students `(talentScore, id)`. Keyset comparison uses Postgres ROW `<` so ties don't drop or duplicate rows across pages. Rate limiting is enforced by `express-rate-limit` in `lib/rate-limit.ts` with three buckets — `authLimiter` (10 / 15 min, keyed by IPv6-normalised IP + email, mounted on `/auth/{register,login,forgot-password,setup-password}` — `setup-password` is this codebase's name for the password-reset finisher; there is no separate `/auth/reset-password` route), `searchLimiter` (60 / min on the four list endpoints, GETs only), and `globalLimiter` (1000 / 15 min process-wide backstop on everything under `/api`). All three emit standardised `RateLimit-*` + `Retry-After` headers on 429. Webhook endpoints sit above the limiters in route mount order so provider retries from Stripe/Paystack can't be throttled. The `app.set('trust proxy', 1)` already in place keeps `req.ip` honest behind the Replit proxy. The previous in-memory `.filter()` passes (kicking in around 5k candidate rows) are gone; `jobMatchesFilters` from `lib/job-filters.ts` is retained for the saved-search digest worker, which still operates on pre-fetched in-memory row sets. Score recomputation in `/jobs/:id/matches`, `/dashboard/candidate/:id`, and `daily-deck.ts` now goes through `createMatchScoreMemo()` (see `lib/matching.ts`) — a **per-request** map keyed on the sorted skill lists + experience + talent score + verified flag. Cross-request caching is intentionally avoided: the memo is allocated inside the handler and garbage-collected with the response. New FK + composite indexes (`applications_candidate_id_idx`, `applications_job_id_idx`, `applications_candidate_status_idx`, `applications_job_status_idx`, `candidate_institutions_candidate_idx`, `candidate_institutions_institution_idx`, `candidate_institutions_inst_dept_idx`) back the new SQL filters so the EXPLAIN plans stay on indexed seeks even at scale. Smoke test: `pnpm --filter @workspace/scripts run loadtest` (autocannon) hits each endpoint for 8s × 8 connections; on an empty session it confirmed ~3000 RPS sustained with p95 < 7 ms (all 401 from `requireAuth`, exercising the limiter + cursor parse path).
- **Drizzle migrations replace `drizzle-kit push` in production**: `lib/db/drizzle/0000_loud_exodus.sql` is the committed baseline generated from the current schema (includes every existing table + the new pagination-supporting indexes). `lib/db/src/migrate.ts` is a tiny `drizzle-orm/node-postgres/migrator` runner wired up as `pnpm --filter @workspace/db migrate` (dev) and `pnpm --filter @workspace/db migrate:prod` (production / post-merge hook). `scripts/post-merge.sh` now runs `migrate` instead of `push`, so a merged task can no longer silently mutate the live schema — schema changes must go through `pnpm --filter @workspace/db generate` and ship as a reviewed SQL file. `push` and `push-force` remain available for local dev convenience only and are not invoked by any automation. The `out` path was set explicitly in `drizzle.config.ts` so generate + migrate use the same `lib/db/drizzle/` folder.
